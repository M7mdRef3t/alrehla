# Project Working Rules (Dawayir-V2)

هذه التعليمات هدفها منع تكرار الشغل أو إضافة حاجات موجودة بالفعل، وتخلي أي تطوير مبني على فهم كامل للمشروع.

## User-Centric Rule (قاعدة أساسية)

- التعامل مع المالك كمستخدم فعلي للمنصة، وكل اقتراح/ميزة/تعديل لازم يبنى مباشرة على احتياجاته الشخصية وتجربته اليومية.
- أي قرار تصميم/منتج لازم يوضح: "ليه ده مفيد للمالك؟" قبل التنفيذ.

## Discovery Pass (إلزامي قبل أي تعديل كبير)

قبل ما تنفذ ميزة/مكون/مودال/سيرفس جديد:

1. اعمل **مسح سريع للمشروع**:
   - ابحث بـ `rg` عن كلمات مفتاحية مرتبطة بالطلب (مثلاً: `Result`, `Modal`, `Plan`, `AI`, `Baseline`, `Symptom`).
   - افحص أماكن التجميع/الربط: `src/App.tsx`, `src/components/AppSidebar.tsx`, `src/state/*`, `src/modules/*`.
2. لو لقيت حاجة مشابهة:
   - **أعد استخدامها** أو **ادمجها** بدل إنشاء نسخة جديدة.
   - لو لازم تعمل جديد، اكتب سبب واضح: "مش موجود" أو "موجود لكن لا يدعم X".
3. راجع وجود أي **Feature flags / Disabled UI**:
   - ابحث عن: `ENABLED_`, `disabled`, `قريبًا`, `VITE_*`.
4. راجع التكرار والتشابه:
   - لا تنشئ نسختين لنفس التجربة (مثلاً: modalين لنفس الوظيفة) إلا لو فيه سبب UX واضح.

## Duplication Guard (قاعدة منع التكرار)

- ممنوع إنشاء مكون/سيرفس جديد قبل التأكد أنه غير موجود.
- عند إضافة أي ملف جديد:
  - لازم يكون له **نقطة استخدام واضحة** (import + render) أو يكون **مؤسس لمرحلة قريبة** ومذكور صراحة أنه غير مفعّل مع سبب.

## Clean-Up Policy

- إذا ظهر أثناء العمل ملف/مكون مكرر:
  - الأفضل: دمج/توحيد (Single Source of Truth).
  - ثم إزالة النسخة القديمة أو ربطها إذا كانت لها قيمة منفصلة.

## AI Integration Policy

- AI "أساسي" لكن لازم يوجد fallback ثابت لا يكسر التجربة.
- لازم أي شاشة/خطة AI توضّح للمستخدم هل المحتوى:
  - `AI` أو `قالب` (Fallback).

## What To Report Back

بعد Discovery Pass، قدّم للمالك:

- "اللي موجود بالفعل" (أين؟)
- "اللي مكرر/غير مفعّل" (أين؟)
- "الخطة المقترحة" (ماذا سنغيّر؟ ولماذا؟)
