# B2B Portal Input Placeholder Logic Flow

## Goal
تحسين تجربة المستخدم في بوابة الأعمال B2B عبر توضيح صيغة المدخلات المطلوبة.

## Mental Model
- المستخدم داخل شاشة إضافة عضو جديد في بوابة الأعمال.
- النظام يحتاج لتوجيه المستخدم بصيغة الكود المطلوب إدخاله (مثل: B2B-123456).
- يجب أن يكون التوجيه واقعي ولا يعتمد على رموز مبهمة مثل XXXXXX لضمان فهم المستخدم الفوري للصيغة الصحيحة.

## Inputs / Outputs
- Inputs: `clientCode` (مثل: B2B-123456) و `clientAlias` من المستخدم عبر واجهة الإدخال.
- Outputs: تحديث قيم state وعرضها بوضوح في الـ UI مع placeholder واقعي.

## States
- `idle`: المستخدم يرى حقول الإدخال مع الـ placeholder الافتراضي "د اع (B2B-123456)".
- `typing`: المستخدم يكتب القيم بناءً على المثال المقدم.
- (تكتمل حالات النجاح والفشل عند الارتباط بالخدمات الخلفية - API).

## Transitions
1. `idle -> typing` عند بدء إدخال النصوص.
2. (يتم التوسع فيها عند ربط النموذج بتقديم الطلب).

## Edge Cases
- إدخال الكود بصيغة غير متوافقة مع `B2B-` أو إدخال حروف بدلاً من الأرقام في الجزء الخاص بالكود الفعلي.

## Failure & Fallback
- في حال كان الكود غير صالح، يجب توجيه رسالة خطأ للمستخدم بناءً على التحقق المسبق أو رد الـ API.
- (حاليًا، التركيز على عرض الواجهة والـ placeholders بشكل صحيح).

## Performance Constraints
- Target complexity: O(1) لتحديث واجهة المستخدم (تغيير نصوص الـ placeholder فقط).
- Max latency: N/A - تغييرات فورية على الـ Frontend.
- Memory constraints: N/A

## Security Constraints
- Validation rules: التأكد من تنقية وتصحيح المدخلات (Sanitization) قبل إرسالها لـ API.
- Authorization boundary: التأكد من صلاحيات المستخدم للوصول لبوابة الأعمال (B2B Portal).
- Sensitive data handling: الأكواد والألقاب لا تعتبر بيانات حساسة بشكل بالغ ولكن تخضع لمعايير الأمان المعتادة.

## Acceptance Criteria
1. عرض نص placeholder كالتالي: "د اع (B2B-123456)" في حقل إدخال الكود.
2. إزالة أي تواجد لرموز الـ "X" في الـ placeholders ضمن بوابة الـ B2B للوضوح.
